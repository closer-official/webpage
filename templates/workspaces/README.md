# テンプレ制作ワークスペース（リポジトリ内）

**本体（運営 SPA・API・既存 LP パイプライン）** と **テンプレ増産用の作業場** を、**同じ Git・同じ Vercel** のまま **ディレクトリで分ける**ための場所です。

**Git / Vercel を初めて触る人**は、先に **`docs/テンプレ制作ワークスペース要件書.md` の「背景知識ゼロ向け」** と **`docs/開発から納品までの流れ.md`** を読んでください。

## 公開ギャラリー掲載テンプレ（フォルダ名は覚えやすいスラッグ）

内部 ID（API・DB）は従来どおり。**Cursor では次のフォルダだけ開く**と作業しやすいです。

| フォルダ | 内部テンプレ ID | 表示名の目安 |
|----------|-----------------|--------------|
| `cafe-multistore/` | `cafe_1` | テンプレ1（複数店舗・ミニマル） |
| `gym-valx/` | `gym_personal_neon` | テンプレ2（CLOSER・ジム販売LP） |
| `navy-cyan-web/` | `navy_cyan_consult` | テンプレ3（ネイビー×シアン・Web/LP） |

各フォルダに **`package.json`** があり、**このフォルダをカレントにしたまま**次が実行できます。

- **`npm run deploy`** → リポジトリルートの `deploy`（コミット・push・本番反映）
- **`npm run build`** → ルートの `build`（型チェック + Vite）

> デプロイ対象は **常にサイト全体**です。テンプレ単体の部分デプロイはありません。編集したファイルがルートのビルドに含まれれば、ギャラリー・該当テンプレの顧客 LP の両方に反映されます。

## 新規テンプレを増やすとき（最短）

1. `templates/workspaces/_template/` をコピーし、  
   `templates/workspaces/<テンプレのスラッグ>/` を作る。
2. その **`<スラッグ>/` フォルダだけ**を Cursor で「フォルダーを開く」（推奨）。
3. `SPEC.md` を埋め、`mock/` でプロトタイプを書く。
4. 仕様が固まったら、**本体取り込み**は `docs/テンプレ制作ワークスペース要件書.md` の **§4 チェックリスト** に従い、通常は **`master` 相当のブランチで1本の PR** にまとめる。

## やらないこと

- この配下だけで完結させようとして **`server/buildHtml.js` を無計画に長期並行編集**しない（競合の温床）。  
  取り込みは **まとめて1回**か、**小さな `server/templates/...` モジュールを先に追加**してから本体に接続する。

## 掲載テンプレの見え方をまとめて見る

- **`/admin/template-hub.html`** … `GET /api/public/template-catalog` を読み、**縮小プレビュー・全画面リンク**を並べる運営用ページ（本番は Basic 認証下の `/admin/`）。
