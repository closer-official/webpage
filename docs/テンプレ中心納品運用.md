# テンプレ中心の納品運用（標準方針）

**この文書が「納品の意味するところ」の正**です。  
アプリ内に複数の「テンプレ」概念が歴史的に混在していても、**運用判断はここだけ見ればよい**ようにします。

---

## 前提（合意したルール）

### 1. 納品済み・これから納品する「商品」には、コードで手を入れない

- **店舗A用だけ HTML を書き換える／店舗専用ブランチを切る**、といったことは **しない**。  
- 不具合や文言の見直しは **テンプレ側**または **購入者が触るデータ（DB上のコンテンツ）** で解決する。

### 2. 今後の納品は「テンプレを用意する」ことに集約する

- 新しい見た目・業種が必要になったら **新規テンプレを1本追加**する（依頼→あなたが実装）。  
- テンプレが揃っていれば、**現場はテンプレ選択だけ**で同じフローを繰り返す。

### 3. すべてのテンプレに「購入者用ページ」を持つ

- 購入者（店側）が **自分で差し替えたい文章・写真**を入力・選択する画面。  
- ここでの編集対象は **コードではなく、保存されるコンテンツデータ**（例: `lpContent` 相当）。

### 4. 標準フロー（全員同じ）

1. **テンプレを選ぶ**（なければ **新規テンプレ作成を依頼**）  
2. **購入者用ページ**で、差し替えたい文章・写真に変更する  
3. 購入者が **「この内容で作成する」**（確定ボタン）を押す  
4. その内容が反映された **サイトが完成**し、**独自URL**（例: `siteKey` 付きサブドメイン）が割り振られる  
5. 必要に応じて **URLの上書き**（独自ドメイン・上位プランのホスト名など）

### 5. なぜこれがよいか（あなたの意図の言語化）

- 一人ひとりに合わせて **コードをいじる**と、追うべき状態が増えて破綻しやすい。  
- **テンプレ＋購入者データ**に分けると、**全員同じ流れ**にできる。  
- 表示不具合が出たら **テンプレを1回直せば、そのテンプレを使っている店はまとめて直る**。店舗を一軒ずつ追わなくてよい。

## 技術的な対応関係（ざっくり）

| 運用での言葉 | リポジトリでのイメージ |
|--------------|------------------------|
| テンプレ | `public/deliverables/{テンプレslug}/` の静的LP ＋ そのテンプレ用の初期JSON形 |
| 購入者用ページ | 既存の店舗向けCMS（例: `gym-lp.html`）や、今後それに相当する **同一パターンの編集UI** |
| 確定後の中身 | DB（またはローカルJSON）の **`lpContent` を siteKey 単位で保存** |
| 独自URL | `siteKey` ＋ DNS（`*.store-official.net` 等）＋ 必要なら独自ドメイン |
| コードで触るもの | **テンプレ追加・テンプレ修正・共通API**のみ。店舗個別の差分は持たない |

※ ダッシュボードに並ぶ **旧「テンプレ1〜15」系**は、**この標準フローとは別レーンの遺産**として扱い、**新規納品の正は上表（deliverables ＋ 購入者ページ）**とする。

## 実装済み（コード）

| 項目 | 内容 |
|------|------|
| テンプレ正の一覧 | `GET /api/product-lp-templates`（公開・ログイン不要） |
| 定義 | `server/productLpTemplates.js` の `STANDARD_PRODUCT_LP_SLUGS` + `META` · 現状は `gym-valx-intro` と `web-closer-intro` のみ（`japanese-history-higashi` は新規標準から除外） |
| 一覧の変更 | 環境変数は使わない。`STANDARD_PRODUCT_LP_SLUGS` と `META`（および deliverables / lpContent）を編集。運営「⓪ デザイン」の案内と揃える |
| 店舗作成（標準） | `POST /api/store-setup/provision` · **clone は上記一覧に限定** · 認証は **管理者Cookie**（ウィザードはログイン済みブラウザ）または API 直叩き時のみ **`Authorization: Bearer <STORE_SETUP_PROVISION_TOKEN>`** |
| ウィザードUI | `/admin/store-wizard.html`（各テンプレ **見た目を見る** → 店舗キー・店舗用ID/PW → 作成 → URL 表示） |
| 運営用（緊急・レガシー） | `POST /api/admin/lp-cms-provision` は従来どおり **全 `LP_CMS` スラッグ**を `cloneFrom` に使用可 |
| web-closer マルチ店舗 | `web-closer-intro` の `index.html` / `admin.html` が **`?site=` または `*.store-official.net`** で `siteKey` を解決（ジムLPと同型） |
| 営業プレビュー | テンプレは **商品一覧に載るスラッグのみ** · `web-closer-intro` 用の Places 下書きマージを追加 |
| デザインタブ | 旧 buildHtml テンプレカードは **既定で非表示** · 表示する場合はボタン、または `VITE_SHOW_LEGACY_BUILDER_TEMPLATES=true` |

## 残りの改善（任意）

1. 「この内容で作成」ボタンを **決済完了メールのリンクだけ**で実行できるよう、Stripe → `STORE_SETUP_PROVISION_TOKEN` のワンタイム化。  
2. 新テンプレ追加時は `productLpTemplates.js` の `META` と `deliverables` と `lpContent.json` の初期JSONを揃える。  
3. 納品済み個別改修の依頼は **テンプレ改修 or 購入者CMS** に寄せ、コードに店舗名を書かない。

以上。
