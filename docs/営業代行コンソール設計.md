# 営業代行コンソール設計（文章化）

営業代行フロー（ログイン制プレビュー、Places API、決済後本番公開、翌月 B 社への一括振込）を実装する際の **画面のたたき台** と **データモデル** を1本にまとめたものです。  
数値（API 50 回など）は **仮** のため、契約・実測に合わせて差し替えてください。

---

## 1. 関係者と用語

| 用語 | 意味 |
|------|------|
| **薫之介** | 事業主・Stripe 受取人・本番公開の最終オペレーター（または自動化） |
| **B** | 営業会社（トップ）。翌月一括振込の相手 |
| **A** | 営業マン（B 所属）。コンソールの主ユーザー |
| **C** | 店舗経営者（顧客） |
| **プレビュー** | 未契約〜契約直後の下書きサイト。`preview.closer-official.com` 配下 |
| **本番** | `店舗名.closer-official.com` / `店舗名.store-official.net` / 独自ドメイン |

## 3. データモデル（エンティティと責務）

### 3.1 図（概念）

```
Organization (B)
    └── SalesRep (A) ──< PreviewSession >── PlaceRef (place_id)
                              │
                              ├── PreviewSnapshot（表示用データ・JSON）
                              └── Sale（決済確定後）
                                      └── ProductionTenant（siteKey, ドメイン）
```

### 3.2 `Organization`（営業会社 B）

| フィールド | 型（例） | 説明 |
|------------|----------|------|
| `id` | UUID | |
| `name` | string | 会社名 |
| `slug` | string | ログイン識別子などに利用可 |
| `created_at` | timestamptz | |

### 3.3 `SalesRep`（営業マン A）

| フィールド | 型（例） | 説明 |
|------------|----------|------|
| `id` | UUID | |
| `organization_id` | FK | B |
| `email` | string unique | ログイン |
| `password_hash` | string | または IdP の subject |
| `display_name` | string | |
| `status` | enum | `active` / `suspended` |
| `created_at` | timestamptz | |

**インデックス**: `organization_id`, `email`

### 3.4 `PlaceRef`（Google `place_id` の正規化）

Places API は **`place_id` がキー**。同一店舗の複数プレビューで Details を共有する。

| フィールド | 型（例） | 説明 |
|------------|----------|------|
| `place_id` | string PK | Google の place_id |
| `details_json` | jsonb | Details のレスポンス（規約・容量に応じてトリム） |
| `details_fetched_at` | timestamptz | キャッシュ TTL 判定用 |
| `photo_refs` | jsonb | 写真参照のメタのみ保持可（実 URL は短命のため） |

**ポリシー**: `details_fetched_at` から **N 日**経過で再取得（N は運用で決定）。  
**ログ**: `PlaceDetailsFetch` を別テーブルに切ってもよい（監査・課金按分用）。

### 3.5 `PreviewSession`（1 回の「プレビュー案件」）

営業が QR で見せる単位。URL は推測困難な **ランダム ID** を推奨。

| フィールド | 型（例） | 説明 |
|------------|----------|------|
| `id` | UUID PK | URL に使う `店舗ID`（例: `preview.closer-official.com/{id}`） |
| `sales_rep_id` | FK | 誰が作ったか |
| `organization_id` | FK | 集計のため冗長でも可 |
| `template_id` | string | テンプレ識別子 |
| `place_id` | string FK | `PlaceRef` |
| `status` | enum | `draft` / `ready` / `checkout_started` / `paid` / `published` / `archived` |
| `snapshot_json` | jsonb | LP に差し込む最終データ（テキスト・画像 URL・営業用コピー） |
| `created_at` | timestamptz | |
| `updated_at` | timestamptz | |

**インデックス**: `sales_rep_id`, `organization_id`, `status`, `created_at`

### 3.6 `ApiUsageEvent`（API コスト連動の根拠）

上限は設けない代わりに **ログは必須**。

| フィールド | 型（例） | 説明 |
|------------|----------|------|
| `id` | UUID | |
| `sales_rep_id` | FK | |
| `organization_id` | FK | |
| `preview_session_id` | FK nullable | 紐づかない呼び出しも記録可 |
| `api_kind` | enum | `autocomplete` / `place_details` / `place_photo` 等 |
| `place_id` | string nullable | |
| `billed_units` | int | 社内換算用（1 Details = 1 などルール化） |
| `estimated_cost_minor` | int nullable | 円の端数は集計テーブルで |
| `created_at` | timestamptz | |

**集計**: 月次で `organization_id` または `sales_rep_id` 単位にロールアップした `ApiUsageMonthly` を別テーブルにしてもよい。

### 3.7 `Sale`（成約）

**「成約1件」のソース・オブ・トゥルース**。

| フィールド | 型（例） | 説明 |
|------------|----------|------|
| `id` | UUID | |
| `preview_session_id` | FK | どのプレビューから申し込んだか |
| `sales_rep_id` | FK | 報酬按分 |
| `organization_id` | FK | |
| `stripe_checkout_session_id` | string unique | |
| `stripe_payment_intent_id` | string nullable | |
| `amount_total_minor` | int | 決済総額（最小通貨単位） |
| `currency` | string | `jpy` 等 |
| `line_items_json` | jsonb | プラン・オプション SKU（原価合算用） |
| `status` | enum | `pending` / `paid` / `refunded` / `disputed` |
| `paid_at` | timestamptz | **API 按分の「成約」基準時刻**（契約で定義） |
| `created_at` | timestamptz | |

**ルール案（契約別紙に明記）**

- **成約に含める API 回数**: `paid_at` 以前に同一 `preview_session_id`（または同一 `place_id`＋同一 `sales_rep_id`）で発生したイベントのみ、など **境界を一文で固定**する。
- **「50 回（仮）」**: **成約 1 件あたり、薫之介負担とみなす `billed_units` の上限**。超過分は B への振込から相殺（または A の歩合から、契約どおり）。

### 3.8 `ProductionTenant`（本番）

既存のマルチテナント設計（`siteKey`、`lpCmsAccounts` 等）と接続。

| フィールド | 型（例） | 説明 |
|------------|----------|------|
| `site_key` | string PK | 本番識別子 |
| `sale_id` | FK unique | どの成約から生えたか |
| `preview_session_id` | FK | コピー元 |
| `primary_host` | string | `xxx.store-official.net` 等 |
| `custom_domain` | string nullable | |
| `published_at` | timestamptz | |
| `cms_username` | string | 既存 provision と整合 |

**フロー**: `ADM-01` 本番公開時に `PreviewSession.snapshot_json` → 本番用 `lpContent` 等へコピーし、`provision` 相当を実行。

## 4. 主要フローとデータの動き

### 4.1 プレビュー生成

1. A が `PREV-01` でテンプレ＋検索 → Autocomplete ログを `ApiUsageEvent` に記録  
2. 候補確定 → Details 取得 → `PlaceRef` 更新 → `PreviewSession` 作成、`snapshot_json` 生成  
3. QR は `PreviewSession.id` の URL

### 4.2 決済

1. C が申込 → Stripe 完了 webhook で `Sale` を `paid` に  
2. `preview_session_id` でプレビューと紐付け、`PreviewSession.status = paid`

### 4.3 本番公開

1. 薫之介が `ADM-01`（または自動）で `ProductionTenant` 作成、DNS  
2. `PreviewSession.status = published`、プレビュー側は閲覧不可 or リダイレクト

### 4.4 翌月 B への振込

1. `Sale` 集計（返金・チャージバック除外ルールを契約で固定）  
2. オプション原価・**薫之介固定取り分（例: 19,900 円）** を line_items から算出  
3. `ApiUsageEvent` を成約ルールに従い按分し、**超過 API 代**を算出  
4. **振込額** = 合意式（例: 決済総額 − 原価合計 − 超過 API − 振込手数料）を B へ

## 5. 値引き・「鉄の掟」とシステム

- **カート計算**: プラン本体は **薫之介取り分の下限を下回らない**ようバリデーション。値引きは **B／A の取り分側のみ**から差し引く設計にするとブレない。  
- Stripe では **line item** と **metadata**（`rep_id`, `org_id`, `preview_session_id`）を残すと後追いしやすい。

---

## 6. セキュリティ・コンプライアンスメモ

- Places の **表示義務（attribution）** をプレビュー・本番テンプレ双方に組み込む  
- プレビューは **robots noindex**  
- サーバーにしか **Google API キー**を置かない（フロントに直載せしない）

## 8. 実装状況（コードベース）

以下を **2025-03 時点** で追加済みです（データは `app_store.salesAgency` または `server/data/json/salesAgency.json`）。

| 項目 | 内容 |
|------|------|
| 営業ログイン | `POST /api/sales/auth/login` · Cookie `sales_rep_session` |
| Places | `POST /api/sales/places/autocomplete` · `details`（7日キャッシュ）· `parse-maps-url` |
| API ログ | `apiUsage` 配列（営業マン別、`places_autocomplete` / `place_details` 等） |
| プレビュー | `POST /api/sales/preview-sessions` → `publicId`（`pv`+24hex）· ジムテンプレ JSON を自動差し込み |
| プレビュー表示 | `GET /api/sales/preview-lp-content/:publicId` · ジムLP `?salesPreview=` で読み込み |
| QR | `GET /api/sales/preview-sessions/:publicId/qr.png` |
| Stripe | `checkout-redirect` に `salesPreviewPublicId` で metadata（`sales_preview_public_id`, `sales_rep_id`, `sales_org_id`） |
| 本番化 | `POST /api/admin/sales-publish-preview`（スナップショット → `siteKey` の `lpContent` + CMSアカウント） |
| 手動成約フラグ | `POST /api/admin/sales-preview-mark-paid`（Stripe Webhook 無しでもステータス更新可） |
| 初回シード | `POST /api/admin/sales-seed-rep`（`orgName`, `repEmail`, `repPassword`, `repDisplayName`） |
| コンソールUI | `/admin/sales-console.html` |
| プレビューホスト | Middleware: `SALES_PREVIEW_HOSTNAME`（既定 `preview.closer-official.com`）の `/pv{24hex}` → ジムLP |

### 環境変数（推奨）

| 変数 | 用途 |
|------|------|
| `GOOGLE_MAPS_API_KEY` | サーバー Places（必須） |
| `SALES_REP_SESSION_SECRET` | 営業Cookie署名（未設定時は `LP_CMS_SESSION_SECRET` 等にフォールバック） |
| `SALES_PREVIEW_PUBLIC_BASE` | QR/Stripe return 用のプレビューURL親。例: `https://preview.closer-official.com`（末尾スラッシュなし） |
| `SALES_CHECKOUT_API_ORIGIN` | Checkout API があるオリジン。プレビューだけ別サブドメインのとき `https://メインのドメイン` を明示 |
| `SALES_PREVIEW_HOSTNAME` | Edge で受けるプレビュー用ホスト名（既定: `preview.closer-official.com`） |
