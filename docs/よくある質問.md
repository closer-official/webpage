# よくある質問

## フルオートのメインの使い方

**フルオート**タブで **地域・カテゴリ・件数** を入れて **開始** すると、サーバーが検索→分析→LP3案+DMまで一括実行し、**ダッシュボード**に載ります。キューを触る必要はありません。

手動で Maps から足した店を処理したい場合のみ **手動・詳細**タブを使います（API 接続時はサーバーキューへ送る／「サーバーへ送る」ボタンあり）。

---

## フルオートで「Google Maps API: INVALID_REQUEST」と出る

**フルオート・サーバー側**は、まず **Places の Text Search（レガシー HTTP）** で検索します。レガシーが **INVALID_REQUEST / REQUEST_DENIED** で結果ゼロのときは、自動で **Places API (New) の `searchText`** に切り替えます（その後の店舗詳細は従来どおり **Place Details**）。ブラウザ用キー（`VITE_GOOGLE_MAPS_API_KEY`）とは **別** に、**`GOOGLE_MAPS_API_KEY`** が必要です。

**よくある原因**

1. **Vercel / server の `GOOGLE_MAPS_API_KEY` に「ウェブサイトのリファラー制限」付きキーを入れている**  
   サーバーからのリクエストにはリファラーが付かないため **INVALID_REQUEST** になります。  
   **対処:** Google Cloud Console で **新しい API キー**を作成し、  
   - **アプリケーションの制限** → **なし**（または IP 固定サーバー向けなら IP）  
   - **API の制限** → **Places API**・**Places API (New)**・**Places Details** など、実際に使う API だけに絞る  
   そのキーを **Vercel の `GOOGLE_MAPS_API_KEY`**（ローカルなら `server/.env`）に設定する。

2. **Places API（レガシー）と (New) の有効化・請求**  
   レガシーだけ無効なプロジェクトでは **New の searchText** にフォールバックしますが、**Places API (New)** の有効化と **請求**が必須です。**Place Details**（詳細取得）も有効にしてください。

3. **キーをコピーしたときの改行・空白**  
   Vercel の環境変数に余計なスペースが入っていないか確認する。

---

## フルオートで「店が見つからない」のに、Maps にはたくさん出る

**仕様です。** フルオートの自動検索は **Google Maps にウェブサイトURLが未登録の店だけ** を集めます（営業リスト用）。東京都港区・渋谷などのカフェ・サロンは **ほぼ全店サイト掲載済み** のため、検索しても **0件** になりやすいです。地方・郊外・商店街寄りのエリアを試すか、「手動・詳細」でサイトありの店をキューに入れてください。

---

## Gemini で「API key was reported as leaked」403

キーが GitHub 等に公開されたと Google に検知された状態です。**そのキーは使えません。** [Google AI Studio](https://aistudio.google.com/) などで **新しい API キーを発行**し、`server/.env` と Vercel の **`GEMINI_API_KEY`** を差し替え、**リポジトリやチャットにキーを貼らない**ようにしてください。

## 「Failed to fetch」と出る

**原因:** フロント（localhost:5173）がバックエンドAPI（localhost:3001）にリクエストを送ったが、届いていない／サーバー側でエラーになっている状態です。

**確認すること:**

1. **サーバーが起動しているか**  
   このアプリは **2つ** 起動が必要です。  
   - **ターミナル1**: プロジェクトルートで `npm run dev` → フロント（http://localhost:5173）  
   - **ターミナル2**: プロジェクトルートで `npm run server`、または `cd server` して `npm run dev` → APIサーバー（http://localhost:3001）  
   「サーバーで収集」「キューを1件処理」を使うときは、**ターミナル2で API サーバーが動いている**状態にし、`Server running at http://localhost:3001` と出ているか確認してください。
2. **フロントの .env に API のURLが書いてあるか**  
   プロジェクト**ルート**の `.env` に次の1行があるか確認し、なければ追加してフロントを再起動してください。  
   `VITE_API_URL=http://localhost:3001`
3. **API キーは「2か所」に分けて書く**  
   - **ルートの .env**（フロント用）  
     `VITE_GOOGLE_MAPS_API_KEY=...` と `VITE_API_URL=...` のみ。Vite がここを読みます。  
   - **server/.env**（サーバー用）  
     `GOOGLE_MAPS_API_KEY=...` と `GEMINI_API_KEY=...` を**必ずこちらに**書いてください。  
     フルオートの「サーバーで収集」「キューを1件処理」は、**サーバー**が Maps と Gemini を呼ぶため、**server/.env** のキーが使われます。ルートの .env にだけ書いてあるとサーバーはキーを読めず、処理が失敗します。
4. **サーバーを再起動する**  
   server/.env を編集したら、サーバー（`npm run dev`）を一度止めてから再度起動してください。

## OKボタンで送信される？共有用URLは？

**OK** は「承認済みリストに入れる」だけで、**メールやLPの送信はしません**。送信は「メール送信（mailto）」や「送信済み」など、別操作で行います。

**共有用URL** … 検閲ダッシュボードの各案件に「共有用URL（スマホ・他端末で開けます）」が表示されます。このURLをコピーしてスマホや別のPCで開くと、同じLPが表示されます。**本番（Vercel等）デプロイ後**はどこからでも開けます。ローカル開発では同じPC内のみ有効です。詳しくは `docs/承認から公開・編集・購入までの流れ.md` を参照してください。

## 作成したウェブページを確認するには

**手段1: 検閲ダッシュボードでプレビュー**

1. **検閲ダッシュボード** タブを開く。
2. 承認待ちの案件の**中央**に、生成されたLPのプレビューが表示されます。
3. 複数案がある場合は「案1」「案2」「案3」のドロップダウンで切り替えられます。
4. **「このページを別タブで開く」** を押すと、そのLPのHTMLが**新しいブラウザタブ**で開きます。スクロールや表示をじっくり確認できます。

**手段2: PDF・テキストから作成した場合**

1. **PDF・テキストから作成** タブで、デザイン選択 → 編集 → プレビュー確認のあと、
2. **「HTMLをダウンロード」** をクリックすると、1つのHTMLファイルが保存されます。
3. そのファイルをダブルクリックしてブラウザで開くと、完成したウェブページを確認できます。

**手段3: ダッシュボードの案件をHTMLで保存したい場合**

検閲ダッシュボードで「このページを別タブで開く」で開いたタブで、ブラウザの「名前を付けて保存」や「印刷→PDFに保存」で保存できます。

## VITE_GOOGLE_MAPS_API_KEY はどこで取得するか

Google Maps Platform の API キーです。次の手順で取得できます。

1. **Google Cloud コンソール**  
   https://console.cloud.google.com/ にアクセスし、Googleアカウントでログインします。
2. **プロジェクトを作成**  
   画面上部のプロジェクト選択 → 「新しいプロジェクト」→ 名前を入力して作成。
3. **API を有効化**  
   「APIとサービス」→「ライブラリ」で次を検索し、それぞれ「有効にする」をクリック。  
   - **Maps JavaScript API**（ブラウザの「Google Maps で取得」用）  
   - **Places API**（同上・店舗検索・詳細取得）
4. **API キーを作成**  
   「APIとサービス」→「認証情報」→「認証情報を作成」→「API キー」を選択。作成されたキーをコピーします。
5. **キーを制限（推奨）**  
   認証情報一覧でそのキーを開き、「アプリの制限」で「HTTP リファラー」を選び、  
   `http://localhost:5173/*` や `https://あなたのドメイン/*` を追加すると、第三者による悪用を抑えられます。
6. **.env に設定**  
   プロジェクトルートの `.env` に次のように書き、保存します。  
   `VITE_GOOGLE_MAPS_API_KEY=ここに貼り付け`  
   保存後、`npm run dev` を再起動してください。

**注意:** Maps / Places API は従量課金です。無料枠がありますが、使用量に応じて料金がかかります。Google Cloud の「請求」で予算アラートを設定しておくと安心です。

## 作成した Maps の API キーでいい？「鍵を表示」は？

**「Maps Platform API Key」が作成されていればそのままでOKです。**

- **「鍵を表示します」** をクリックすると、実際のキー文字列（`AIza...` で始まる）が表示されます。  
  その文字列をコピーし、プロジェクトルートの `.env` に  
  `VITE_GOOGLE_MAPS_API_KEY=貼り付けたキー`  
  と書いて保存してください。
- **制限** に「32 個の API」などと出ていれば、Maps 系の API が有効になっています。  
  このツールでは **Maps JavaScript API** と **Places API** が有効であれば十分です。

## 「同意画面を構成してください」って何？設定しないとダメ？

**OAuth（オース）同意画面**は、**ユーザーが Google でログインし、あなたのアプリに「自分のデータへアクセス」を許可する**ときに表示される画面です。

- **Maps の API キーだけで地図・店舗検索を使う場合**  
  → ユーザーが Google にログインする必要はなく、**同意画面の設定は必須ではありません**。  
  警告が出ていても、**今の用途（マップ表示・Places 検索）だけなら無視して大丈夫**です。
- **あとから Gmail やカレンダーなど「ユーザーのデータ」に触れる API を入れる場合**  
  → そのときに「同意画面を構成」が必要になります。

まとめ: **マップと Places だけ使うなら、同意画面は設定しなくてOK**です。

## マップ以外に、AI（Gemini）用の API キーも必要？

**はい。フルオート（口コミ分析・DM文面の自動作成）を使うなら、別途「Gemini 用の API キー」を用意します。**

- **Maps の API キー**（今作成したもの）  
  → Google Cloud の「認証情報」で作ったキー。  
  → ブラウザの「Google Maps で取得」や、サーバーでの店舗検索に使う。

- **Gemini（AI）の API キー**  
  → **別の場所**で発行します。  
  → **Google AI Studio**: https://aistudio.google.com/apikey  
  → ここで「API キーを作成」し、発行されたキーをコピー。  
  → **サーバー**の `.env`（`server/.env`）に次のように書く:  
  `GEMINI_API_KEY=貼り付けたキー`

**使い分け:**

| やりたいこと           | 使うキー                          | 設定する .env              |
|------------------------|-----------------------------------|----------------------------|
| ブラウザで Maps 検索   | Google Cloud の Maps API キー     | ルートの `.env`（VITE_GOOGLE_MAPS_API_KEY） |
| サーバーで Maps 収集   | 同上（同じキーでOK）              | `server/.env`（GOOGLE_MAPS_API_KEY） |
| 口コミ分析・DM自動作成 | Gemini の API キー                | `server/.env`（GEMINI_API_KEY） |

マップだけ使うなら Maps のキーだけで十分。**フルオート（キューを1件処理）まで使うなら、Gemini のキーも server/.env に追加**してください。

## 学習を開始すると INVALID_REQUEST やエラーが出る

**原因:** 学習は**サーバー側**で Google Maps Places API（Text Search）を呼びます。本番（Vercel）では **GOOGLE_MAPS_API_KEY** を環境変数に設定する必要があります（VITE_GOOGLE_MAPS_API_KEY はブラウザ用のため、学習では使われません）。

**確認すること:**

1. **Vercel の環境変数**に **GOOGLE_MAPS_API_KEY** を追加しているか（Settings → Environment Variables）。
2. **Google Cloud 側**で、そのキーに **Places API**（と **Places API (New)** の Text Search が必要な場合あり）が有効か。キー制限で「HTTP リファラー」だけにしているとサーバーからのリクエストは拒否されるため、**サーバー用は「IP アドレス」制限にするか、制限なしで別キーを発行**する。
3. **請求（Billing）**が有効か。Places API は従量課金のため、プロジェクトで請求を有効にしていないと INVALID_REQUEST になることがある。

## 無名・これから立ち上げの店を自分で見つけたとき、どうやってウェブページを作るか

**このツールでの流れ:**

1. **キュー** タブを開く。
2. **「手動で追加」** を使う。  
   店舗名・住所・メモ（Instagram URL や「〇〇で見つけた」など）・業種を入力して「キューに追加」。
3. キュー一覧でその店の **「調査」** をクリック。
4. 調査画面で **コンセプト・強み・イメージカラー（テンプレート）** を入力し、「LP作成してダッシュボードへ」。
5. **検閲ダッシュボード** でLPプレビューとDM文面を確認し、OK/NG。

Google Maps やサーバーでの自動収集は使わず、**自分で見つけた1件を手動で登録してLPを作る**使い方です。サーバー（フルオート）を止めていても、この手動フローはブラウザだけで動きます。

## インスタなどをAIが見て、雰囲気や文章を組むことはできるか

**現状の仕様:**

- **Google Maps の口コミ** … サーバーで「キューを1件処理」すると、Maps の口コミを取得し、Gemini が雰囲気・客層・コンセプト・強みを分析してLP案とDM文面を自動作成します。
- **Instagram** … インスタの投稿内容を**自動で取得してAIに渡す機能はまだありません**。**写真数枚のリンクを自動取得してLPに使う**ことも、APIの制限と利用規約の関係で行っていません。  
  理由の一つは、Instagram の公式APIはビジネス用途が中心で、一般の「投稿をまとめて分析」には制限があるためです。スクレイピングは利用規約違反になるためおすすめしません。

**できること（いまのツールで）:**

- インスタで見つけた店を **「手動で追加」** のメモ欄に **InstagramのURL** や **キャプションの抜き書き** を貼っておく。
- **調査** の「コンセプト」「強み」に、**自分でインスタの雰囲気や文言を要約して入力**する。  
  → その内容でLPが生成されます（AIは使わず、入力どおりにテンプレートに載せます）。

**将来的に可能なこと:**

- 「Instagram のキャプション（またはURL）を貼り付ける」→ そのテキストを Gemini に渡して**雰囲気・訴求文を自動で要約・生成する**機能を追加することは技術的には可能です。  
  その場合は「テキストを貼る」形式にし、規約に触れない形にします。  
  画像をAIで解析する場合は、画像用のAPI（Gemini の画像入力など）の利用を検討することになります。

## 写真・画像はどうやって選んで持ってくるか（ウェブページがない店の場合）

### Google Maps の写真を自動で使う（推奨）

**キューに「Place ID」がある店**（マップ検索で「キューに追加」した店・サーバーで収集した店）は、**キューを1件処理**すると **Google Maps に登録されている写真を最大6枚まで自動取得**し、次のように割り当てます。

- **1枚目** … ヒーロー画像・OGP画像
- **2枚目以降** … コンセプト・メニュー・ギャラリーなど、上から順にセクションの画像
- **A-1（Minimal Luxury）テンプレ** … 複数枚をヒーローでスライド表示

**条件:** その店が **Google マップの「写真」タブに写真を登録していること**。登録がなければ従来どおりデフォルト画像 or 手動URLになります。

**手動で追加した店**（Place ID なし）では写真の自動取得はできません。その場合は下記の「手動でURLを貼る」で対応してください。

### Instagram の写真を自動取得することはできるか

**できません。** Instagram の公式APIはビジネス用途が中心で、「店の投稿を数枚まとめて取得してLPに使う」ような一般利用には制限があり、利用規約にも触れやすいため実装していません。インスタで見つけた店の雰囲気は、**手動で追加**のメモや調査画面のコンセプト・強みに文言で入力し、画像は **Maps の写真（マップから追加した場合）** または **ストック写真・自前URL** で補う形になります。

### 手動で画像URLを入れる場合

#### ヒーロー画像・OGP画像（1枚）

- **どこで設定するか**  
  **SEO設定**の「**OGP画像URL**」に入れたURLが、LPの**ヒーロー**と**OGP画像**に使われます。
- **何を入れるか**  
  **画像の直リンク（URL）**を1本入れます。画像アップロード機能はないため、次のいずれかでURLを用意して貼り付けます。
  - **無料ストック写真** … [Unsplash](https://unsplash.com/) や [Pexels](https://www.pexels.com/) で画像の**直接URL**をコピーして貼る。
  - **自前ホスト** … 自社サーバー・S3・Imgur などにアップロードした画像の**公開URL**を貼る。
- **何も入れない場合**  
  （Maps の自動取得もないときは）**テンプレートごとのデフォルト画像**（Unsplash の1枚）がヒーローに使われます。

#### セクションの画像

- **どこで設定するか**  
  **ページ内容の編集**で、各セクションの「**画像URL**」欄に入れます。
- **自動で入る場合**  
  Maps から写真を取得した場合は、上から順にセクションへ自動で割り当て済みです。足りない分や差し替えは手動で編集できます。

### まとめ

| 状況                         | 写真の入り方                                               |
|------------------------------|------------------------------------------------------------|
| マップから追加 → 1件処理     | **Google Maps の写真を最大6枚まで自動取得**して hero・セクションに割り当て |
| 手動で追加 → 1件処理         | 自動取得なし。デフォルト画像 or **OGP画像URL・セクション画像URLを手動で貼る** |
| 既存案件の編集               | **SEO設定「OGP画像URL」** と **ページ編集「画像URL」** で差し替え可能 |

**注意:** Maps の写真を使う場合、生成されたHTMLに Google Place Photo のURL（APIキーを含む場合あり）が含まれます。本番でHTMLを配布する場合は、Google Cloud で API キーの制限（参照元URLなど）を設定することを推奨します。
