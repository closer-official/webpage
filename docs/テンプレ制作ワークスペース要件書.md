# テンプレ制作ワークスペース — コピペ用要件書（Cursor 向け）

この文書は **新しい Cursor ウィンドウ** を開くとき、**チャットの最初に貼る**／**プロジェクト指示（Rules / AGENTS）に貼る**用です。  
**同じ Git リポジトリ・同じ Vercel デプロイ**のまま、**本体コードと触るファイルを分けて**テンプレだけ量産するための要件です。

> **読者の前提**  
> **人間でも AI でも「このリポジトリのことを一切知らない状態」**から、次の **ゴール**に到達するための地図です。  
> 用語（Git / Vercel）は **§ 背景知識ゼロ** で補います。

---

## ゴール定義（何を達成すればよいか）

### ゴール A: テンプレートギャラリーに載せる

- **本番 URL:** [テンプレートギャラリー（Closer）](https://webpage.closer-official.com/template-gallery)  
- **ページの仕組み:** ブラウザが **`GET /api/public/template-catalog`** で一覧 JSON を取り、各テンプレの **`previewUrl`**（例: `/api/template-preview/<id>`）で **縮小プレビュー iframe** を表示している。
- **載る条件（データの流れ）:**  
  1. **`server/templateRegistry.js`** のビルトイン一覧に **ID が載っている**（または DB に **公開ステータスのカスタムテンプレ**がある）。  
  2. **`server/templateCatalogMeta.js`** にその ID の **カテゴリ・タグ・人気**がある（ビルトインは必須に近い）。  
  3. **`renderTemplatePreview` → `buildHtml`** でその ID の HTML が生成できる（通常 **`server/buildHtml.js`** と、プレビュー一致用の **`src/lib/buildHtml.ts` + `src/lib/templates.ts`**）。

→ 下記 **「§4a ギャラリー掲載までの実装順」** をすべて満たし、**`npm run build` が通る状態で本番ブランチにデプロイ**されれば、[テンプレートギャラリー](https://webpage.closer-official.com/template-gallery) に **既存カードと同じ扱い**（一覧・週間ピックアップ候補・縮小 iframe・別タブ全画面）で現れる。

### ゴール B: 「販売・納品」に使える形にする（店舗を増やす標準フロー）

ギャラリーに載っている **ビルトイン buildHtml テンプレ**だけでは、**必ずしも**「店舗ウィザードで選んで `cloneFrom` 納品できる商品」とは **別レーン**です。

このリポジトリでいう **新規納品の標準**は **`docs/テンプレ中心納品運用.md`** のとおり、ざっくり次です。

| 必要なもの | 置き場の例 |
|------------|------------|
| 購入者が編集する **固定 LP 本体** | `public/deliverables/<slug>/`（静的 HTML ＋ CMS 用 JS 等） |
| **ウィザード・プロビジョンで選べる商品**として登録 | **`server/productLpTemplates.js`** の **`STANDARD_PRODUCT_LP_SLUGS`** と **META**（slug・名前・編集画面パス・デモパス） |
| 初期文面の形 | `server/data/json/lpContent.json` 等（slug ごと） |

→ **店舗を増やして販売する形式**にするには **§4b** を満たす。**ギャラリーだけ欲しい**なら **§4a のみ**でよい。

### ゴール A と B の関係（よくある誤解）

- **同じ見た目でも ID が別**なことがある（例: ギャラリーは **`gym_personal_neon`**（buildHtml）、納品商品は **`gym-valx-intro`**（deliverables）のように **別 slug**）。  
- **「ギャラリーにも載せて、納品商品にもしたい」** なら **両方のチェックリスト**を意識する。仕様は **`SPEC.md` の `INTEGRATION_TYPE`** に書き分ける。

## 背景知識ゼロ向け（Git・Vercel・このプロジェクト）

### このプロジェクト（`webpage`）は何か

- **1 つのウェブアプリのソースコード**が入ったフォルダ群です（フロント・サーバー・LP テンプレなど全部ここ）。
- フォルダの **どこかの親階層に `.git` という隠しフォルダ**があり、その範囲が **「1 つの Git リポジトリ」** です。  
  - 通常、クローンした直下が `webpage` で、その中に `server/` や `src/` や `templates/` があります。

### Git（ざっくり）

- **Git** = ソースコードの **変更履歴を記録する仕組み**です。
- **リポジトリ** = その履歴の入ったプロジェクト全体のこと（この場合は `webpage` の Git）。
- **クローン** = GitHub などから **自分の PC に同じリポジトリをコピーした状態**。あなたが編集しているのは **そのコピー上のファイル**です。
- **コミット** = 「この時点の変更を履歴に残す」操作。  
- **プッシュ（`git push`）** = その履歴を **GitHub などの共有場所に送る**操作。  
- **重要:** テンプレ作業用に **`webpage` を別の場所にもう一度コピーして新しいフォルダで始めない**でください。コピー先に **別の `.git` がない／古い**と、どちらが本番に載るコードか分からなくなります。**常に「チームが使っている 1 つのクローン」の中**で作業します。

### Vercel（ざっくり）

- **Vercel** = このプロジェクトのようなサイトを **インターネット上で動かすホスティング**の一つです。
- 多くの場合 **GitHub の特定ブランチ（例: `master`）に新しいコミットが来たら自動でビルド・公開**する設定になっています。
- つまり流れはざっくり **`PC で編集 → コミット＆プッシュ → Vercel がビルド → 本番 URL が更新されうる`** です。
- **テンプレ作業だけの人**は、最初は **Vercel の画面を触らなくてよい**です。  
  **「変更をリポジトリに載せる」担当**が `npm run deploy` や `git push` を行います（手順は `docs/開発から納品までの流れ.md`）。

### 「テンプレ用の別 Cursor プロジェクト」とは何をするか

- **新しい Git や新しい Vercel プロジェクトを作ることではありません。**
- **同じ `webpage` クローンの中**の、次のフォルダだけを Cursor で開き直すイメージです。  
  `templates/workspaces/<スラッグ>/`  
- **公開ギャラリー掲載の既存3テンプレ**は、覚えやすいフォルダ名で用意済みです（内部 ID は従来どおり）。  
  `cafe-multistore`（`cafe_1`）／`gym-valx`（`gym_personal_neon`）／`navy-cyan-web`（`navy_cyan_consult`）。各フォルダの `package.json` で **`npm run deploy`** ＝ リポジトリルートのデプロイ（本番全体が更新）です。
- こうすると **本体の巨大ファイルを誤って壊しにくい**・**競合しにくい**、というメリットがあります。

### 自分が正しい場所にいるか確認する（チェックリスト）

1. エクスプローラーまたはターミナルで、**`webpage` のルート**に `.git` がある（または `git status` が使える）。
2. 作業フォルダは **`webpage/templates/workspaces/...`** の **内側**である（リポジトリの外にフォルダを作っていない）。
3. 「デスクトップに webpage のコピーをもう一つ作った」状態になっていない。

## 0. 前提（必須・要約）

- 作業対象は **`webpage` リポジトリのクローン**であること（上記「背景知識ゼロ」を満たすこと）。
- **別パスにリポジトリを複製してテンプレだけそこで進めない**（Git・Vercel と本体がずれる）。
- テンプレ専用の作業場所はリポジトリ内の次の配下に置く:  
  **`templates/workspaces/<あなたのスラッグ>/`**

---

## 1. 初回だけ：フォルダの作り方（0 → コーディング開始）

リポジトリルートで、テンプレ用スラッグ（例: `salon-dark-2025`）を決め、**雛形をコピー**する。

```text
templates/workspaces/_template/  →  templates/workspaces/<スラッグ>/
```

コピー後、必ず **`SPEC.md`** のプレースホルダを埋める（テンプレ ID 案・表示名・用途）。

**Cursor で開くフォルダ**

- **推奨:** `ファイル` → `フォルダーを開く` →  
  `.../webpage/templates/workspaces/<スラッグ>`  
  こうすると、このフォルダ配下だけに注意を向けやすく、本体の巨大ファイルを誤って壊しにくい。
- **代替:** リポジトリルートを開いたまま **マルチルートワークスペース** に `templates/workspaces/<スラッグ>` を追加。

## 2. このワークスペースの責務（やること）

| やる | 内容 |
|------|------|
| ✅ | 見た目・構成のプロトタイプ（`mock/` 内の HTML / CSS / 画像メモ） |
| ✅ | テンプレ ID・表示名・カテゴリ案・キーワード案の整理（`SPEC.md`） |
| ✅ | セクション構成・コピー案・ナビ項目のメモ（`notes/`） |
| ✅ | 本体に取り込むときの **チェックリストを自分で全部チェック**（下記 §4） |

---

## 3. このワークスペースではやらないこと（ネック回避）

| やらない | 理由 |
|----------|------|
| ❌ 本体の `server/index.js` をテンプレのために大きく書き換えない | 他ブランチ・他タブと競合しやすい |
| ❌ `server/buildHtml.js` を「とりあえず全部ここで」いじらない | ファイルが巨大で競合の中心。取り込みは **マージ担当が1回**にまとめるか、**テンプレ専用の小さなモジュールを切り出してから** 1 ファイルに追記する運用が安全 |
| ❌ 別リポジトリを新規作成して「あとでマージ」 | Git/Vercel が本体と一致しなくなる |

※ 最終的な **ビルトイン登録** と **buildHtml への実装** は、テンプレが固まったタイミングで **本体側ブランチで1本化**する。

## 4. 本体への取り込みチェックリスト（要約）

ビルトインとして **ギャラリー・ヒアリング**に出すときの項目一覧。  
**背景ゼロで実装するときは必ず §4a の順序どおり**に進めること（順を飛ばすとプレビュー 404 や一覧欠落になる）。

1. [ ] `server/templateRegistry.js` … **ID・表示名**（**正**はここ）
2. [ ] `server/templateCatalogMeta.js` … ギャラリー用カテゴリ・タグ・人気スコア
3. [ ] `server/buildHtml.js` … サーバー側 HTML 生成
4. [ ] `src/lib/buildHtml.ts` + `src/lib/templates.ts` … **プレビューと本番の見た目を一致**
5. [ ] （該当する場合）`src/types.ts` の `STYLES`、推定ルール `server/inferTemplatePriority.js` / `src/lib/inferTemplateFromSearch.ts`
6. [ ] `npm run build` が通る

**deliverables 系の固定 HTML 商品**（`public/deliverables/*`）として出す場合は、別ラインのため `docs/テンプレまわりの置き場とデプロイ.md` の表を読み、**商品定義**（例: `server/productLpTemplates.js`）も忘れずに。

### §4a ギャラリー掲載までの実装順（背景ゼロ・機械的手順）

**達成基準:** [テンプレートギャラリー](https://webpage.closer-official.com/template-gallery) に **新しいカードが並び**、**「別タブで全画面」**で `/api/template-preview/<あなたのID>` が開き、**ヒアリング**（`/customer-intake` 系）のテンプレ選択にも **同じ ID** が出る（ビルトインの場合）。

#### データの流れ（読まなくてもよいが、迷ったらここ）

1. ギャラリーページ HTML は `server/templateGalleryPage.js` が返すが、**テンプレ一覧の中身はクライアントが** `GET /api/public/template-catalog` で取得（実装: `server/publicTemplateCatalog.js` の `buildPublicTemplateCatalog`）。
2. その API は `server/templatePreview.js` の **`getTemplateCandidates`** を呼ぶ。**ビルトイン**は **`server/templateRegistry.js` の `BUILTIN_BUILD_HTML_TEMPLATES`** から来る。
3. 各カードの `previewUrl` は **`/api/template-preview/<id>`**（`server/index.js` のルート）。中身は **`renderTemplatePreview`**（`server/templatePreview.js`）→ 最終的に **`buildHtml`**（`server/buildHtml.js`）で HTML 文字列。

→ **レジストリに ID がないと catalog に載らない。buildHtml に実装がないとプレビューが壊れる。**

#### 実装する順番（この順でやる）

| 順 | ファイル | やること |
|----|----------|----------|
| 1 | （決定） | **テンプレ ID** を1つ決める（英小文字＋アンダースコア推奨。既存の `cafe_1` などと被らない）。**表示名（日本語）**も決める。 |
| 2 | `server/templateRegistry.js` | `BUILTIN_BUILD_HTML_TEMPLATES` に `{ id, name }` を **1行追加**。ここが **一覧の正**。 |
| 2b | `server/data/json/galleryDraftBuiltins.json` | （任意だが推奨）新規 id を `draftBuiltinIds` に追加 → **一般ギャラリー・ヒアリングでは非表示**のまま実装検証。完成後は **`/admin/template-hub.html` の「ギャラリーに公開」**、またはこの配列から id を削除。 |
| 3 | `server/templateCatalogMeta.js` | `BUILTIN_CATALOG_META` に **同じ id キー**で `category` / `categories` / `popularity` / `tags` を追加（ギャラリーのチップ・並びに効く）。 |
| 4 | `server/buildHtml.js` | その `id` で **完全な HTML が組み立てられる**ように分岐・セクションを実装（分量が大きいのはここ）。 |
| 5 | `src/lib/buildHtml.ts` | **同じ id** でクライアント側も同様に組み立て（プレビューと本番のずれ防止）。 |
| 6 | `src/lib/templates.ts` | **`template-<id>`** 用の CSS を追加（既存テンプレのパターンをコピーして改変）。 |
| 7 | （任意） | キュー・検索から自動でこのテンプレを選ばせたいなら `server/inferTemplatePriority.js` と `src/lib/inferTemplateFromSearch.ts`。運営 UI のドロップダウンなら `src/types.ts` の `STYLES`。 |
| 8 | 検証 | リポジトリルートで **`npm run build`** が成功すること。 |
| 9 | デプロイ | **`docs/開発から納品までの流れ.md`** に従い、共有リモートへ push（例: `npm run deploy`）。本番は Vercel がビルドした結果が [ギャラリー](https://webpage.closer-official.com/template-gallery) に反映される。 |

#### ローカルでの確認（デプロイ前）

- サーバー起動後（例: `npm run server` など、このリポジトリの標準手順に従う）:
  - **`GET /api/public/template-catalog`** の JSON の `templates` に **自分の `id`** があるか。
  - **`GET /api/template-preview/<id>`** が **200 で HTML** を返すか（縮小 iframe の中身）。
- 運営向けに一覧したい場合: **`/admin/template-hub.html`**（カタログ同期の縮小プレビュー・全画面リンク。本番は Basic 認証下の `/admin/`）。

#### 本番での確認（デプロイ後）

- [テンプレートギャラリー](https://webpage.closer-official.com/template-gallery) を開き、**検索・カテゴリ**で自分のテンプレが見えるか、**別タブ全画面**が開くか。

#### ギャラリー以外の「同様の扱い」

- **ヒアリングフォーム**のテンプレ選択も、ビルトインは **`getTemplateCandidates`** 由来なので、**§4a の 2〜4 ができていれば**（レジストリ＋プレビュー可能）、**通常は自動で選択肢に載る**（`intake_bespoke` など別系統を除く）。

#### カスタムテンプレだけギャラリーに載せる場合（参考）

- 管理者が **`/api/template-customizations/save`** 等で **公開（published）** したスキン／ブループリントは **`getTemplateCandidates`** に乗り、**ビルトインと並んで**ギャラリーに出る。  
- この文書の **ワークスペース `mock/`** は主に **ビルトイン追加前のプロトタイプ**向け。カスタムだけならレジストリは触らず運営画面経由も可（詳細は `docs/テンプレまわりの置き場とデプロイ.md`）。
