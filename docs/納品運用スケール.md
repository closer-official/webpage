# 納品が数十〜数百になっても運用を変えずに回す考え方

**「納品の意味・テンプレと購入者の流れ」の正**は **`docs/テンプレ中心納品運用.md`** を先に読んでください（コードで店舗個別に手を入れない・テンプレ修正で一斉修正、など）。

**開発〜本番へのコード反映**は **`docs/開発から納品までの流れ.md`**（標準: `master` ＋ `npm run deploy`）を参照。

## いまの `npm run deploy` は変わる？

**基本、変えません。**

いまの `scripts/deploy.js` は次のような流れです。

1. `git add` → 変更があれば `commit` → `git push`（いまのブランチ、多くは **`master`**）
2. Vercel の **Production Branch** をそのブランチに合わせていれば、**この push で本番も更新**。末尾の **Vercel CLI 本番はデフォルトオフ**（詳細は `docs/開発から納品までの流れ.md` / `docs/本番ブランチ運用.md`）。

これは **「コードをリモートに載せる」** 手順で、店舗が何件あっても **同じ型**で回せます。

このドキュメントで言っている「運用を変えない」とは、

- ❌ 店舗が増えるたびに **デプロイのやり方を別物にする**  
ではなく、  
- ✅ 店舗が増えても **納品時にやる作業の型**（DNS＋APIでアカウント作成）を **いつも同じ**にする  

という意味です。

---

## 結論（推奨の標準形）

**1つの本番デプロイ（1つの Vercel プロジェクトなど）に、店舗は足していく。**  
店舗の違いは **DNS（サブドメイン）＋ `siteKey` ＋ DB（`lpContent` / `lpCmsAccounts`）** で表現し、**納品のたびにデプロイ単位を増やさない。**

こうすると「納品が100件になってもやることの型」は同じです。

| やること | 毎回の納品 |
|----------|------------|
| サブドメイン | `店舗キー.store-official.net` を DNS で同じ本番に向ける |
| 店舗アカウント | `POST /api/admin/lp-cms-provision`（全体ADMIN）で1件作成 |
| 中身 | 店舗管理者が `/admin/gym-lp.html` から編集（または運営が初期だけ投入） |
| コード更新 | **あなたがタイミングを決めて** 本番を1回デプロイ（全店舗共通） |

「店舗ごとに Git ブランチ／プロジェクトを増やす」必要は **基本ありません**。

---

## 「固定したい不安」との両立

- **コードの“勝手な更新”**を避けたい場合は、Vercel の **Production Branch** を **作業しないブランチ**に限定するか、**プレビューだけ別ブランチ**にする、など。**`production` ブランチ運用**は任意（手順は **`docs/本番ブランチ運用.md`**）。標準は **`master` ＋ `npm run deploy`**。
- **意図しない互換破壊**は、**LP向け API と `lpContent` の形**を変えたときに起きます。  
  - 対策: `docs/管理者画面CMS.md` の API を **後方互換**で変える／Breaking は CHANGELOG と移行期間。

「納品1件＝コミット固定」は **VIP向けオプション**として残し、標準運用では使わない、が数百件では現実的です。

## 変えずに済む「納品チェックリスト」（コピペ用）

1. **店舗キー**を決める（英小文字・数字・ハイフン、2〜64文字、先頭末尾は英数字）  
   - 例: `shibuya-gym`
2. **DNS**: `shibuya-gym.store-official.net` → いつもと同じ本番（Vercel）
3. **プロビジョニング**（運営）  
   - **標準:** ブラウザで **`/admin/store-wizard.html`**（テンプレ選択 → `POST /api/store-setup/provision`）  
   - **レガシー・緊急:** `POST /api/admin/lp-cms-provision`（`cloneFrom` に `japanese-history-higashi` 等も可）  
   いずれも `siteKey`, `username`, `password`, `cloneFrom` を指定
4. **店舗管理者へ渡すもの**  
   - 公開URL: `https://shibuya-gym.store-official.net/`  
   - 管理: `https://shibuya-gym.store-official.net/admin/gym-lp.html`  
   - ID / PW
5. （任意）**閲覧数・文言**は店舗側で確認

## 本番環境（Vercel）で共通して1セットだけ持つもの

- Supabase 接続（`SUPABASE_URL` / キー）
- 全体運営 `ADMIN_USERNAME` / `ADMIN_PASSWORD`（プロビジョニング用）
- **`LP_CMS_SESSION_SECRET`**（店舗CMSクッキー用・本番はランダム長文推奨）

店舗数に比例して増やすのは **基本なし**（店舗別ID/PWは DB に蓄積）。

---

## リリース時（本番デプロイ時）の最低限ルール

開発が進んでも手順を変えないために、**デプロイ前に毎回同じ確認**だけ足す。

1. 代表テンプレで LP が開く（例: `gym-valx-intro` またはテスト用 `siteKey`）
2. `GET /api/lp-content/:siteKey` が期待する JSON 構造を壊していない（可能なら CI でスキーマ軽チェック）
3. 店舗CMSログイン → 保存が通る

---

## まとめ

- **数百納品でも運用を変えない**なら、**マルチテナント1本＋プロビジョニング型**を標準にする。  
- **納品のたびに増やすのは DB のレコードと DNS** であり、**デプロイ先やリポジトリの数は増やさない**。  
- 「壊れたくない」は **本番ブランチの運用**と **LP API の後方互換**で守る。

詳細APIは `docs/管理者画面CMS.md` を参照。

---

## テンプレを大量に増やすとき（本体と作業場所を分ける・任意）

**店舗納品の数が増えても**「1 デプロイ・1 リポジトリ」はそのまま維持しつつ、**Cursor で触る範囲**だけ分ける運用ができます。

- **本体:** リポジトリルート（いまの `webpage`）で API・デプロイ・レジストリ取り込みを行う。  
- **テンプレ専用:** 同一リポジトリ内の **`templates/workspaces/<スラッグ>/`** だけを別ウィンドウで開き、プロトタイプと仕様メモに専念する。  
- **Git / Vercel:** 増やさない（いつも同じ clone・同じプロジェクトに push）。

手順・コピペ用の要件全文は **`docs/テンプレ制作ワークスペース要件書.md`**、フォルダ索引は **`templates/workspaces/README.md`** を参照。
