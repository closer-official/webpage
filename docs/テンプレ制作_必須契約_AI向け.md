作業者・AI・オーナー（あなた）の違いは下表のとおり。**迷ったらこのファイルを正**にしてください。

## 1. スコープの約束（必須）

### 1.1 色・テーマの可変はしない

- **店舗向け・社内向けのどちらの UI も、色パレットやアクセント色を「選択式で変える」前提にはしない。**
- テンプレの配色は **テンプレ実装時に CSS で固定**する。運用で変えるのは **文章・URL・写真（画像 URL）などデータ側**に限定する。

### 1.2 雰囲気は写真・ビジュアルで変える

- **ヒーロー・セクション画像・カタログ画像など、画像 URL を差し替え可能なフィールド**（例: `PageContent.heroSlides`、業種ごとの画像配列）を主な差別化手段とする。
- 新規テンプレでは、**「色を変えなくても店ごとに別物に見える」**よう、**写真の置き場所と枚数**を設計に含める。

## 3. アクセシビリティ・品質（最低限）

- **スキップリンク**（既存テンプレ群のパターンに合わせる）。
- 装飾でない画像には **意味のある `alt`**（データ側で渡せる設計にするか、テンプレでプレースホルダを明記）。
- **キーボード操作・フォーカス**が破綻しないこと（特にドロワー・ボタン）。
- **色に依存しすぎない情報設計**（コントラストはテンプレ CSS 側で固定済みの範囲内で担保）。

## 4. Git・チーム運用（「あなたの PC に上げる」ではない）

**正はリモート 1 つ**（GitHub 等）。個人のマシンにだけコードを集める運用にはしない。

1. **オーナー**がリポジトリへ **Collaborator 招待**（または Organization メンバー追加）を行う。  
2. **作業者**は `git clone` で同じリポジトリを取得し、作業前に `git pull` で最新にする。  
3. **作業者**は **`master` / `main` に直接 push しない**のが安全。`feature/テンプレ名` などのブランチを切る。  
4. 実装が終わったら **`git push`** でリモートに載せ、**プルリクエスト（PR）** を出す。  
5. **オーナー**（またはデリゲート）が PR をレビューし、**セクション 2 のチェックリスト**で突き合わせてからマージする。  
6. マージ後、チームは `git pull` で取り込む。  
7. **本番反映**はチームの運用に従う（多くの場合 Vercel が既定ブランチへのマージで自動デプロイ）。

**AI だけに任せて終わりにしない:** 最終的には **人間が PR をレビュー**し、必須契約に沿っているか確認する。

## 5. Cursor（または他 AI）に渡すもの・出だし

### 作業者が Cursor で始めるとき

1. このリポジトリを **clone 済み**のフォルダで Cursor を開く。  
2. **新規チャットの 1 通目**に必ず次を含める:  
   - `@docs/テンプレ制作_必須契約_AI向け.md`（このファイル）  
   - やりたいこと（業種・参考 URL・内部 ID 案・ギャラリーに載せるか 等）  
3. 実装後、AI に **「セクション 2 の表 A〜I（と J）を満たしているか一覧で答えよ」** と依頼するか、自分で突き合わせる。  
4. 変更は **ブランチにコミット**し、**push → PR**。

### オーナーが外注・部下に依頼するときに送る文例（コピペ可）

> リポジトリ [URL] を clone してブランチを切って作業してください。  
> Cursor を使う場合は、**新しいチャットの最初**に `@docs/テンプレ制作_必須契約_AI向け.md` を添付してから要件を書いてください。  
> 終わったら PR を出してください。マージはこちらで行います。

## 6. 明示的にやらないこと（この契約下）

- エンドユーザー向け・社内向けフォームでの **色テーマ選択 UI** の新設を前提にしたテンプレ設計。
- **実行時に任意 CSS を注入**して見た目を変える仕様（セキュリティ・再現性の理由で対象外）。
- **`public/deliverables/**` の既存納品物の改変**（顧客からの明示依頼がない限り。プロジェクトの protected-paths ルール参照）。
