# テンプレまわりの置き場とデプロイ

## 1. なぜファイルが分かれているか

**全部を1ファイルに「統合」することは、現状のアーキテクチャでは現実的ではありません。**

| 役割 | 主な置き場 | 理由 |
|------|------------|------|
| **運営が選べるビルトイン3種の ID・名前** | `server/templateRegistry.js` | **ここが正**。`templatePreview.js` と `conceptTemplates.js` はここを参照する。 |
| **LPのHTMLを組み立てる（サーバー）** | `server/buildHtml.js` | テンプレごとの分岐が巨大。Node 上でだけ動かす。 |
| **同じLPのプレビュー（ブラウザ）** | `src/lib/buildHtml.ts` + `src/lib/templates.ts` | Vite/React 用。TS + クライアント向け。 |
| **deliverables（固定HTML・CMS付き）** | `public/deliverables/*` + `server/productLpTemplates.js` | buildHtml 系とは別ラインの商品。 |
| **初期文面・ショーケース** | `src/data/showcasePresets.ts` | JSON に近いデータ量。 |
| **レガシー用のコンセプトCSS断片** | `server/conceptTemplates.js` | 歴史的経緯で肥大化。段階的整理対象。 |

Node の `buildHtml.js` とフロントの `buildHtml.ts` を機械的に1本化するには、**共有パッケージ**や**ビルドでJSON/CSSを生成するスクリプト**が必要になります。いまは「**ID・名前の正は `templateRegistry.js`**」「**見た目の実装はサーバー/クライアントで二重**」と割り切るのが運用しやすいです。

## 2. `npm run deploy` の結果はどこで見るか

`npm run deploy` は **`scripts/deploy.js`** が動き、だいたい次の流れです。

1. `git add .`
2. 変更があれば `git commit`（メッセージは引数または `Deploy`）
3. **現在のブランチを `git push`**（`origin`）

**見る場所:**

- **ターミナル** … 成功/失敗はそのまま表示されます（`プッシュ完了。` など）。
- **Git ホスト（GitHub 等）** … リポジトリの該当ブランチの最新コミットが更新されているか。
- **Vercel を Git 連携している場合** … その push に紐づく **デプロイが Vercel 側で自動開始**されます。結果は **[Vercel Dashboard](https://vercel.com/dashboard)** の該当プロジェクト → **Deployments** で確認できます（ログ・URL・成功/失敗）。

**注意:**

- 既定では **`vercel --prod`（CLI）は自動では走りません**（事故防止）。Git 連携で十分ならそのままでよいです。
- **標準**: Vercel の **Production Branch** を **`master`**（実際に push しているブランチ）に合わせる → **`npm run deploy` だけで本番まで更新**。
- **任意**: `production` ブランチを挟む運用なら **`npm run deploy:prod`**（`docs/本番ブランチ運用.md`）。

リポジトリが Git で初期化されていない環境では `deploy` は途中で失敗します（`git status` / `remote` 未設定など）。

## 3. 変更するときのチェックリスト（ビルトイン3種）

1. `server/templateRegistry.js` … ID・表示名  
2. `src/types.ts` の `STYLES`（フロントのドロップダウン等）  
3. `src/lib/inferTemplateFromSearch.ts` / `server/inferTemplatePriority.js` … キーワード→テンプレ推定  
4. 見た目を変えるなら `server/buildHtml.js` と `src/lib/templates.ts` の**両方**（ずれるとプレビューと本番が一致しなくなる）
