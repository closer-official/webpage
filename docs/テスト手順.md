# テスト方法と手順

## 前提

- **手動テスト** … ブラウザとサーバーを起動し、画面操作で動作確認する方法です。
- **自動テスト** … 単体テスト（Jest/Vitest）や E2E（Playwright など）は未導入です。必要に応じて後から追加できます。

---

## 1. 環境の準備

### 1-1. フロントのみ（サーバーなし）

```bash
# プロジェクトルート
npm install
npm run dev
```

- ブラウザで http://localhost:5173 を開く。
- `.env` に `VITE_GOOGLE_MAPS_API_KEY` を設定すると「Google Maps で取得」が使える（未設定でも他は動く）。

### 1-2. フロント ＋ サーバー（フルオート用）

**ターミナル1（サーバー）**

```bash
cd server
cp .env.example .env
# .env を編集:
#   GOOGLE_MAPS_API_KEY=...
#   GEMINI_API_KEY=...
npm install
npm run dev
```

- 起動ログに `Server running at http://localhost:3001` が出ればOK。

**ターミナル2（フロント）**

```bash
# プロジェクトルート
npm run dev
```

- **VITE_API_URL は省略可**です。`vite.config` の **プロキシ**で `/api` が自動的に `http://127.0.0.1:3001` に届きます。別ポートの API を使うときだけ `.env` に `VITE_API_URL=...` を書いてください。
- 開いたら最初のタブが **フルオート**。**必ず別ターミナルで `npm run server`（API 3001）を起動**してから **開始** を押してください。API が止まっていると 500/502 になります。

- http://localhost:5173 を開く。
- キュー画面に「フルオート（サーバーで実行）」が出ていればAPI連携できている。

### 学習・自動処理の流れ

- **学習タブ**: 業種と件数を選んで「学習を開始」→ 収集・メタ・デザイン・分析が自動実行。完了後に「学習結果」でテンプレ作成の参考を確認できる。
- **キュー自動処理**: キュー画面で「キューを自動で全件処理（開始）」を押すと、キューが空になるまで約20秒間隔で 調べる→作成→メール文 を自動実行。停止は「自動処理を停止」。
- **ダッシュボード**: レビュー待ち → OK → 「メール送信（mailto）」またはコピーで送信準備 → 「送信済み」で記録。

## 2. 手動テスト手順

### 2-1. サーバーなし（手動フロー）

| # | 確認項目 | 手順 | 期待結果 |
|---|----------|------|----------|
| 1 | タブ切り替え | 「キュー」「検閲ダッシュボード」「PDF・テキストから作成」「設定」をクリック | それぞれの画面が表示される |
| 2 | 手動でキューに追加 | キュー → 手動で追加 → 店舗名を入力して「キューに追加」 | キュー一覧に1件増える |
| 3 | 調査フォーム | キューで「調査」をクリック → コンセプト・強み・イメージカラーを入力 → 「LP作成してダッシュボードへ」 | モーダルが閉じ、検閲ダッシュボードに1件表示される |
| 4 | 検閲ダッシュボード | ダッシュボードで左（店舗情報）・中央（LPプレビュー）・右（DM文面）を確認 | 内容が表示される。DMを編集して「OK」または「NG」を押すとステータスが変わる |
| 5 | PDFから作成 | 「PDF・テキストから作成」→ PDFを選択 or テキスト貼り付け or 空のページから作成 | ステップ2（デザイン選択）に進む |
| 6 | デザイン・編集・エクスポート | デザインを選び「編集・プレビューへ進む」→ 内容・SEOを編集 → 「HTMLをダウンロード」 | プレビューが変わり、HTMLファイルがダウンロードされる |
| 7 | 設定（AI予算） | 設定タブで「AI利用上限」に数値を入れて「保存」 | 保存した値が表示される |

**Google Maps で取得（オプション）**

- `.env` に `VITE_GOOGLE_MAPS_API_KEY` を設定した状態で、キュー → 「Google Maps で取得」に検索ワード（例: 港区 カフェ）を入力 → 「検索」 | サイト未登録の店が一覧表示され、「キューに追加」でキューに追加できる |

### 2-2. サーバーあり（フルオート）

| # | 確認項目 | 手順 | 期待結果 |
|---|----------|------|----------|
| 1 | オプション | 設定タブを開く | 「生成オプション」のスイッチ（多言語・問い合わせフォーム・Instagram/LINE・Presented by・QRコード）が表示され、切り替えで保存される |
| 2 | サーバーで収集 | キュー → フルオートで「港区 カフェ」など入力 → レビュー条件を選択 → 「サーバーで収集」 | エラーでなければ「added: N」件と表示され、キュー一覧が更新される（Maps API キー必須） |
| 3 | キューを1件処理 | 「キューを1件処理（分析→LP3案＋DM）」をクリック | 処理中表示のあと、検閲ダッシュボードに1件増える（Gemini API キー必須）。LPは「案1」「案2」「案3」で切り替え可能 |
| 4 | ダッシュボード（API） | 検閲ダッシュボードで 案1〜3 を切り替え、DMを編集し「OK」または「NG」 | プレビューが切り替わり、OK/NG でステータスが変わり一覧から消える（承認待ちのみ表示のため） |
| 5 | 手動追加（API） | キュー → 手動で追加で1件追加 | サーバーのキューに追加され、一覧に表示される |
| 6 | キュー削除（API） | キュー一覧で「削除」をクリック | その1件がキューから消える |
| 7 | 料金・Stripe（API） | 設定タブで「料金・お支払い」を表示 → 「料金を再計算」 | オプションに応じた合計金額と内訳が表示される。`STRIPE_SECRET_KEY` を設定している場合は「Stripeで支払う」で Checkout に遷移する（テスト時は `sk_test_...` を使用） |

### 2-3. API の直接確認（任意）

サーバー起動中に、ブラウザや curl で次のように確認できます。

```bash
# オプション取得
curl http://localhost:3001/api/options

# キュー取得
curl http://localhost:3001/api/queue

# ダッシュボード取得
curl http://localhost:3001/api/dashboard
```

- 各レスポンスが JSON で返れば、サーバーとストアは動作しています。
- 収集は `POST /api/collect`、処理は `POST /api/process-next` なので、フロントから「サーバーで収集」「キューを1件処理」で試すのが安全です。

## 3. よくある失敗と確認ポイント

| 現象 | 確認すること |
|------|------------------|
| 「フルオート」や「生成オプション」が出ない | `.env` に `VITE_API_URL=http://localhost:3001` があるか。フロントを再起動したか。 |
| 「サーバーで収集」でエラー | サーバーが起動しているか。`server/.env` に `GOOGLE_MAPS_API_KEY` が正しく設定されているか。 |
| 「キューを1件処理」でエラー | `server/.env` に `GEMINI_API_KEY` が設定されているか。キューに1件以上あるか。 |
| ダッシュボードに何も出ない | 承認待ち（status: pending）だけ表示するため、OK/NG 済みは出ません。もう1件「キューを1件処理」するか、キューに手動追加→調査でLP作成。 |
| LPプレビューが真っ白 | テンプレートのCSSや content が不正でないか。ブラウザの開発者ツールでコンソール・ネットワークを確認。 |

## 4. 自動テストを追加する場合（参考）

- **フロント単体** … Vitest + React Testing Library で、各コンポーネントの表示やボタン動作をテスト。
- **API** … 同じく Vitest や Jest で、Express のルートを `request(app)` のように呼び出してステータスコードとJSONを検証。
- **E2E** … Playwright で「キューで手動追加 → 調査 → ダッシュボードでOK」まで流す。

導入する場合は、まず「設定の保存」「キュー追加」「オプション取得」など、影響範囲の小さいところからテストを足すと安全です。
