# 本番ブランチを分ける（`production`）— **任意**

> **標準運用**は **`master` で開発し `npm run deploy` で push**、Vercel の Production Branch を **`master`** に合わせることです（`docs/開発から納品までの流れ.md`）。  
> 以下は **`production` を挟みたいときの参考**です。

**日常の開発〜本番反映〜店舗納品までの全体フロー**のメイン説明は `docs/開発から納品までの流れ.md` を参照してください。

## まず期待値：「事故をほぼ防げる？」

**かなり減らせますが、万能ではありません。**

| 防げやすいこと | まだ起こりうること |
|----------------|-------------------|
| `main` にプッシュしただけで **意図せず本番コードが変わる** | `production` に **間違ったマージ**をしてプッシュする |
| 作業中のブランチをそのまま本番に載せる（Git連携のみの場合） | **`DEPLOY_USE_VERCEL_CLI=1` で `vercel --prod`** を実行すると、**今開いているフォルダの中身が本番に直載せ**される（ブランチと無関係） |
| | 環境変数・Supabase・DNS を誤操作 |
| | 依存パッケージの脆弱性対応で急いで上げた結果の不具合 |

つまり **「本番に載るタイミングを人間が明示的に選べる」** のが主な効果です。

## やること（初回セットアップ）

### 1. Vercel ダッシュボード（※ 2025年頃から場所が変わっています）

**Production Branch は「Git」タブには出ません。** 公式ドキュメントどおり、次の場所です。

1. プロジェクトを開く → **Settings** → **Environments**
2. **Production** を開く
3. **Branch Tracking**（ブランチの追跡）で、本番用ブランチ名を **`production`** に変更（デフォルトは多くの場合 `main`）
4. **Save**

> **エラー `Branch 'production' not found`** のとき: GitHub 側にまだ `production` が無い状態です。下記「Git に `production` ブランチを作る」を **先に実行して `git push origin production` したあと**、もう一度 Save してください。

参考: [Deploying Git Repositories - Production branch](https://vercel.com/docs/deployments/git#production-branch)

**Settings → Git** にあるのは主に LFS・Deploy Hooks・Verified Commits などで、**本番ブランチの指定は Environments 側**です。

これで **`production` ブランチへの push だけ**が「本番デプロイ」になります（Git 連携の場合）。  
`main` への push は **プレビュー**扱いになり、本番用ドメインは追跡中のブランチのデプロイに紐づきます。

### 2. Git に `production` ブランチを作る（初回だけ・Vercel より先に実施）

リポジトリのメインブランチが `master` の場合は `main` を `master` に読み替えてください。

```bash
git checkout main
git pull origin main
git checkout -b production
git push -u origin production
```

以後、**本番に出したいときだけ** `production` を更新します。

## 日々の流れ

### 開発・検証（いつも）

- `main` で作業 → `npm run deploy`  
  - 変更をコミットして **`main` を push** するだけ（本番ブランチ運用時は末尾の **Vercel CLI 本番はデフォルトオフ**）

### 本番に反映するときだけ

**先に開発ブランチ（`main` または `master`）をリモートへ push しておく**（`npm run deploy` など）→ そのあと:

```bash
npm run deploy:prod
```

- `origin/main` があるときはそれを、無ければ `origin/master` を `production` にマージして **`production` を push**（`DEPLOY_SOURCE_BRANCH` で上書き可）
- Vercel が **本番**をビルド・デプロイ

（マージメッセージは引数で渡せます）

```bash
npm run deploy:prod -- "LP CMS 店舗別認証を本番へ"
```

## `vercel --prod` について（重要）

`npx vercel --prod` は **今のディレクトリのファイル**を本番に載せます。**どの Git ブランチかは見ません。**

本番ブランチ運用と **両方**やると、

- `main` にいるまま `vercel --prod` → **本番が main の中身になる**

という事故が起きます。

そのため **デフォルトでは `npm run deploy` の最後に CLI 本番を走らせません。**  
どうしても CLI で本番に載せたい場合だけ:

```bash
set DEPLOY_USE_VERCEL_CLI=1   # Windows cmd
# PowerShell: $env:DEPLOY_USE_VERCEL_CLI="1"
npm run deploy
```

このドキュメントの手順に従う場合: 本番は **`npm run deploy:prod`（= `production` を push）**。  
**リポジトリ標準は `master` ＋ `npm run deploy` のみ**（`docs/開発から納品までの流れ.md`）で、`production` 分離は不要なら読み飛ばしてよい。

## トラブル時

- **本番が古い**: `production` が最新の `main` を取り込んでいるか確認  
- **プレビューは出るが本番が動かない**: Vercel の Production Branch が `production` になっているか確認  
- **意図しない本番更新**: 誰が `production` に push / `vercel --prod` したか（Git ログ・Vercel の Deployments）

## `production1` / `production2` のように複数本番ブランチは？

**1つの Vercel プロジェクトにつき、「本番（Production）」として追跡できる Git ブランチは基本的に 1 本です。**  
`production1`・`production2`・`production3` を **同時にすべて“公式本番URL”** に割り当てる、という分け方は **標準UIではできません**。

やりたいこと別の例:

| 目的 | 現実的なやり方 |
|------|----------------|
| ステージング用URLで別ラインを試す | `staging` など **プレビュー用ブランチ**＋[ブランチにドメインを割り当てる](https://vercel.com/docs/domains/working-with-domains/assign-domain-to-a-git-branch)（ドキュメントの「複数プレビュー段階」のパターン） |
| 顧客ごとに完全に別デプロイ | **Vercel プロジェクトを分ける**（プロジェクトA＝顧客A の本番） |
| チームで段階を増やす | **Pro** の [Custom Environments](https://vercel.com/docs/deployments/environments#custom-environments) |

マルチテナント（店舗は `siteKey` で分ける）なら、**本番ブランチは `production` 1 本**のまま運用するのが一般的です。
