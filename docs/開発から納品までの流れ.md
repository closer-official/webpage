# 開発から納品までの流れ（`master` ＋ `production` 運用）

**納品の思想（テンプレのみコードで持つ・店舗個別は触らない・購入者ページで差し替え）**は **`docs/テンプレ中心納品運用.md`** を参照。

このリポジトリは **開発用ブランチが `master`**、**お客様が見る本番は `production`** を追跡する想定です（Vercel の Production → Branch Tracking = `production`）。

---

## まず全体像（3つのレイヤー）

| レイヤー | 何をするか | ブランチ / 場所 |
|----------|------------|-----------------|
| **① コード開発** | テンプレ追加・ツール改修・バグ修正 | 基本 **`master`** |
| **② 本番デプロイ** | ①を顧客向けサーバーに反映 | **`production`** を更新（`master` をマージ） |
| **③ 店舗ごとの中身** | 文言・画像・キャンペーン・店舗アカウント | **Supabase（DB）**＋DNS。コードデプロイとは別作業 |

「テンプレを増やす」は **①**、「納品」は多くの場合 **②（必要なとき）＋③（毎回）** です。

---

## ① 日常の開発（テンプレを増やす・機能を足す）

1. **`master` にいることを確認**  
   `git checkout master && git pull origin master`

2. **作業**  
   - 例: `public/deliverables/` や `src/lib/templates.ts`、サーバーAPI などを編集

3. **リモートへ載せる**（いつもの）  
   ```bash
   npm run deploy
   ```  
   - 変更をコミットして **`master` を push**  
   - 本番ブランチ運用では **末尾の `vercel --prod` はデフォルトオフ**（本番URLはまだ変わらない）

4. **動作確認**  
   - Vercel の **プレビューURL**（`master` 向けのデプロイ）で確認する  
   - またはローカルで `npm run dev` / `npm run server`

**ここまででは、本番ドメイン（`production` 追跡の本番）は原則まだ古いまま**です。

---

## ② 本番に出す（「納品用のコードを本番サーバーに載せる」）

**お客様向けURLで動いているプログラムを更新したいときだけ**実行します。

1. **`master` の変更をリモートに push 済み**であること（先に `npm run deploy` など）

2. **本番ブランチへ反映**  
   ```bash
   npm run deploy:prod
   ```  
   - `origin/master`（または `origin/main` があればそちらを優先）を **`production` にマージ**して **`production` を push**  
   - Vercel が **本番デプロイ**を走らせる（Branch Tracking = `production` のとき）

3. **マージメッセージを付けたいとき**  
   ```bash
   npm run deploy:prod -- "ジムLP CMS 店舗別認証など"
   ```

> **注意**: 本番は **全店舗で同じコード**が動きます。ここで載せた変更は、同じVercelプロジェクトの **すべての納品先**に効きます。

---

## ③ 納品する「商品」を作る（店舗ごと）

ここでは **コードではなく「その店のLPとアカウント」** を用意します。**毎回デプロイは必須ではありません**（既に本番に欲しいコードが載っていれば、③だけで新店舗を増やせます）。

### 3-1. 決めること

- **サイトキー** `siteKey`（英小文字・数字・ハイフン、2〜64文字）  
  例: `shibuya-gym`  
  → 本番URL例: `https://shibuya-gym.store-official.net`

### 3-2. DNS

- `店舗キー.store-official.net` を **いつもの本番プロジェクト** に向ける（既存と同じVercel）

### 3-3. 店舗用CMSアカウントの作成（運営）

全体管理者（`ADMIN_USERNAME` / `ADMIN_PASSWORD`）でログインできる状態で:

- `POST /api/admin/lp-cms-provision`  
  またはアプリから `api.provisionLpCmsStore({ siteKey, username, password, cloneFrom: 'gym-valx-intro' })`

→ **パスワードはDBにハッシュで保存**（Vercelに店舗数の環境変数は不要）

### 3-4. 店舗へ渡すもの

- 公開: `https://{siteKey}.store-official.net/`  
- 管理: `https://{siteKey}.store-official.net/admin/gym-lp.html`  
- ログインID / パスワード

### 3-5. 店舗側

- 管理画面でキャンペーン・画像URL・申込URLなどを編集・保存（中身は `lpContent` に保存）

## 流れを例でつなぐ

### パターンA: 新テンプレを開発して、あるタイミングで全本番に反映

1. `master` で実装 → `npm run deploy` で push → プレビューで確認  
2. 問題なければ `npm run deploy:prod` → **全店舗の土台コードが更新**

### パターンB: コードは触らず、新店舗だけ納品

1. DNS ＋ `lp-cms-provision` ＋ 店舗へURL/ID/PW連絡  
2. **`npm run deploy:prod` は不要**（本番コードはそのまま）

### パターンC: バグ修正を急いで本番へ

1. `master` で修正 → `npm run deploy`  
2. すぐ `npm run deploy:prod`  
3. 全店舗に修正が乗る（影響範囲を意識）

## コマンド早見表

| やりたいこと | コマンド / 操作 |
|--------------|-----------------|
| 開発をリモートに載せる | `npm run deploy`（`master` push） |
| 本番にコードを反映 | `npm run deploy:prod` |
| 旧CLIで本番直載せ（非推奨） | `DEPLOY_USE_VERCEL_CLI=1 npm run deploy` |
| 新店舗の口座作成 | `POST /api/admin/lp-cms-provision` |
